## 第一章 实验概述

### 1.1 实验背景

纽约市出租车是城市公共交通体系的重要组成部分，每日承载数百万次出行需求。NYC Taxi & Limousine Commission（TLC）监管下的Yellow Taxi（黄色出租车）和Green Taxi（绿色出租车）分别服务于曼哈顿核心区域及外围行政区，构成了纽约市独特的双层出租车服务体系。通过对大规模出行数据的深入分析，可以揭示城市交通的时空规律，为交通管理、运力调度和政策制定提供数据支撑。

### 1.2 实验目标

1. 掌握大规模交通数据的获取、清洗与预处理方法
2. 运用多种数据分析方法（描述性统计、聚类分析、回归预测等）挖掘数据价值
3. 设计并实现一个交互式数据可视化大屏系统
4. 基于数据分析结果提出有针对性的运营建议

### 1.3 实验环境

| 项目 | 说明 |
|:---|:---|
| **操作系统** | Windows |
| **编程语言** | Python 3.x |
| **核心库** | pandas, numpy, scikit-learn, pyshp |
| **数据格式** | Parquet（列式存储，高效读取） |
| **可视化框架** | ECharts 5（前端图表库） |
| **3D渲染** | Three.js（3D出租车模型展示） |
| **Web框架** | Flask（轻量级HTTP服务器） |
| **天气数据** | NOAA NCDC（美国国家海洋和大气管理局） |

### 1.4 实验流程

```
数据获取 → 数据清洗 → 特征工程 → 统计分析 → 聚类分析 → 回归预测 → 可视化设计 → 系统实现
```

## 第三章 数据清洗规则与方法

### 3.1 清洗策略概述

数据清洗采用**多阶段标记-过滤**策略：对每条记录逐一检查，标记不合规的原因，最终只保留通过所有检查的记录。清洗脚本为 `clean_data.py`，对Yellow和Green两类数据分别执行相同的清洗规则。

### 3.2 清洗规则详解

共设计了 **15条清洗规则**，按检查顺序如下：

#### 规则1：去除重复记录

- **规则**：基于所有原始列进行完全重复检测，保留首次出现的记录
- **原因**：数据采集过程中可能产生重复记录
- **方法**：`df.duplicated(keep='first')`

#### 规则2：年份过滤

- **规则**：上车时间（pickup_datetime）的年份必须为2018年
- **原因**：数据文件可能包含跨年数据
- **方法**：`pd.to_datetime(pickup_col).dt.year != 2018`

#### 规则3：时间逻辑校验

- **规则3a**：下车时间必须晚于上车时间（`dropoff > pickup`）
- **规则3b**：行程时长不得低于30秒（`duration >= 30s`）
- **规则3c**：行程时长不得超过6小时（`duration <= 6h`）
- **原因**：时间逻辑异常说明数据记录有误；过短行程可能是误操作；过长行程可能是计费系统故障
- **参数**：`MIN_DURATION_SEC = 30`, `MAX_DURATION_SEC = 21600`

#### 规则4：乘客数量校验

- **规则**：乘客人数必须在1-6人之间
- **原因**：NYC出租车法定最大载客量为5人（加1名儿童），0人或超过6人属于异常值
- **参数**：`MIN_PASSENGERS = 1`, `MAX_PASSENGERS = 6`

#### 规则5：行程距离校验

- **规则**：行程距离必须大于0且不超过200英里
- **原因**：距离为0说明未实际行驶；超过200英里在NYC范围内不合理
- **参数**：`MAX_DISTANCE = 200`

#### 规则6：票价金额校验

- **规则6a**：票价（fare_amount）必须大于0
- **规则6b**：票价不得超过$500
- **原因**：票价为0或负数说明数据异常；超过$500的极端值可能为系统错误
- **参数**：`MAX_FARE = 500`

#### 规则7：总金额校验

- **规则**：总金额（total_amount）必须大于0
- **原因**：总金额为0或负数说明该行程无效

#### 规则8：支付方式校验

- **规则**：支付方式仅保留信用卡（type=1）和现金（type=2）
- **原因**：本实验仅分析信用卡和现金两种主要支付方式，其他类型（无费用、争议等）不具统计意义
- **参数**：`VALID_PAYMENT_TYPES = {1.0, 2.0}`

#### 规则9：费率代码校验

- **规则**：RatecodeID必须在1-6范围内
- **原因**：NYC TLC定义的标准费率代码为1-6，超出范围的值属于异常
- **参数**：`VALID_RATECODES = {1, 2, 3, 4, 5, 6}`

#### 规则10：金额字段非负校验

- **规则**：extra、mta_tax、tip_amount、tolls_amount、improvement_surcharge等金额字段不得为负数
- **原因**：各项附加费用为负数属于数据异常

#### 规则11：固定附加费校验

- **规则11a**：MTA税必须等于$0.50
- **规则11b**：改进附加费必须等于$0.30
- **原因**：2018年NYC TLC规定的固定附加费标准为MTA税$0.50和改进附加费$0.30，不符合标准值的记录可能存在数据错误
- **参数**：`EXPECTED_MTA_TAX = 0.50`, `EXPECTED_IMPROVEMENT_SURCHARGE = 0.30`

#### 规则12：附加费取值校验

- **规则**：extra字段只能取$0.00、$0.50或$1.00
- **原因**：NYC出租车附加费仅在这三个值之间取值（分别对应无附加、高峰时段、夜间时段）
- **参数**：`ALLOWED_EXTRAS = {0.0, 0.50, 1.00}`

#### 规则13：小费比例校验

- **规则**：小费金额不得超过票价的100%
- **原因**：小费超过票价属于极端异常值
- **参数**：`MAX_TIP_RATIO = 1.0`

#### 规则14：过路费校验

- **规则**：过路费不得超过$100
- **原因**：NYC区域过路费最高约为$15-20，超过$100属于异常
- **参数**：`MAX_TOLLS = 100`

#### 规则15：系统性缺失值检测

- **规则**：如果关键字段（store_and_fwd_flag、RatecodeID、passenger_count、payment_type、trip_type）中有3个或以上为空值，则删除该记录
- **原因**：多个关键字段同时缺失说明该记录整体数据质量极差

### 3.3 清洗参数汇总

| 参数名 | 值 | 说明 |
|:---|:---:|:---|
| MAX_FARE | $500 | 最大合理票价 |
| MAX_DISTANCE | 200 mi | 最大合理行程距离 |
| MAX_DURATION_SEC | 21,600 s (6h) | 最大合理行程时长 |
| MIN_DURATION_SEC | 30 s | 最小合理行程时长 |
| MAX_PASSENGERS | 6 | 最大乘客人数 |
| MIN_PASSENGERS | 1 | 最小乘客人数 |
| VALID_PAYMENT_TYPES | {1, 2} | 有效支付方式 |
| VALID_RATECODES | {1, 2, 3, 4, 5, 6} | 有效费率代码 |
| EXPECTED_MTA_TAX | $0.50 | 标准MTA税 |
| EXPECTED_IMPROVEMENT_SURCHARGE | $0.30 | 标准改进附加费 |
| ALLOWED_EXTRAS | {$0, $0.50, $1.00} | 有效附加费 |
| MAX_TIP_RATIO | 1.0 (100%) | 最大小费比例 |
| MAX_TOLLS | $100 | 最大过路费 |

### 3.4 清洗流程代码逻辑

```
对每个月份文件：
  1. 读取Parquet文件
  2. 初始化 _clean_flag = 'keep'
  3. 依次执行15条规则，不符合的标记为 'remove' 并记录原因
  4. 过滤保留 _clean_flag == 'keep' 的记录
  5. 删除辅助列和全空列
  6. 保存为 _clean.parquet 文件
```

## 第四章 数据预处理与特征工程

### 4.1 数据预处理流程

数据预处理脚本为 `data_prep.py`，负责从清洗后的数据中提取、计算和聚合所有图表所需的数据。

#### 4.1.1 全局统计量计算

遍历所有月份的清洗后Parquet文件，累加计算：

- 总出行量（total_trips）
- 总票价（total_fare）→ 计算平均票价
- 总距离（total_distance）→ 计算平均距离
- 总时长（total_duration）→ 计算平均时长
- 各月出行量（monthly_counts）

#### 4.1.2 分层采样

由于全年数据量超过1亿条，直接加载全部数据进行详细分析（如聚类）不现实。因此采用**分层采样**策略：

- **采样量**：200,000条
- **分层方式**：按月份和车型均匀分配（每月每种车型约 `200000 / (12 * 2) ≈ 8,333` 条）
- **随机种子**：`random_state=42`（保证可复现性）
- **缓存机制**：采样结果缓存为 `_sample_cache.parquet`，避免重复采样

#### 4.1.3 时间特征提取

从上车时间中提取以下特征：

| 特征名 | 提取方式 | 说明 |
|:---|:---|:---|
| _month | `dt.month` | 月份（1-12） |
| _hour | `dt.hour` | 小时（0-23） |
| _day | `dt.day` | 日期（1-31） |
| _dow | `dt.dayofweek` | 星期几（0=周一, 6=周日） |
| _duration_min | `(dropoff - pickup).total_seconds() / 60` | 行程时长（分钟） |

## 第五章 分析方法与算法原理

### 5.1 描述性统计分析

#### 5.1.1 方法说明

对核心数值字段（票价、距离、时长、乘客数等）计算基本统计量：

- **集中趋势**：均值（mean）、中位数（median）
- **离散程度**：标准差（std）、四分位距（IQR = Q3 - Q1）
- **分布形态**：最小值（min）、最大值（max）、百分位数（P5, P25, P75, P95）
- **频次统计**：按月份、小时、区域、支付方式等维度分组计数

#### 5.1.2 应用场景

- KPI卡片（Chart 1）：全局均值
- 月度趋势（Chart 5）：按月分组计数
- 24小时分布（Chart 6）：按小时分组计数
- 票价箱线图（Chart 9）：五数概括（min, Q1, median, Q3, max）
- 距离分布（Chart 8）：直方图 + 百分位数

### 5.2 相关性分析

#### 5.2.1 方法说明

计算数值字段之间的 **Pearson相关系数**，构建相关矩阵：

$$r_{xy} = \frac{\sum(x_i - \bar{x})(y_i - \bar{y})}{\sqrt{\sum(x_i - \bar{x})^2 \cdot \sum(y_i - \bar{y})^2}}$$

分析字段包括：trip_distance、fare_amount、tip_amount、tolls_amount、total_amount、passenger_count、extra、mta_tax、improvement_surcharge、duration_min。

#### 5.2.2 应用场景

- 字段相关性热力图（Chart 3）：识别字段间的线性关系，为后续建模提供特征选择依据

### 5.3 K-Means聚类分析

#### 5.3.1 算法原理

K-Means是一种无监督学习算法，将数据划分为K个簇，使得簇内平方和（WCSS）最小化：

$$\min \sum_{k=1}^{K} \sum_{x_i \in C_k} \|x_i - \mu_k\|^2$$

其中 $C_k$ 为第k个簇，$\mu_k$ 为该簇的质心。

#### 5.3.2 实现细节

| 参数 | 值 | 说明 |
|:---|:---|:---|
| K（簇数） | 4 | 基于出行特征的自然分层 |
| 特征变量 | trip_distance, fare_amount, duration_min | 距离、票价、时长 |
| 数据预处理 | StandardScaler标准化 | 消除量纲差异 |
| 初始化 | n_init=10 | 多次随机初始化取最优 |
| 随机种子 | random_state=42 | 保证可复现性 |
| 采样量 | 50,000条 | 从分层采样中二次抽样 |

#### 5.3.3 数据预处理

聚类前对数据进行以下过滤：

- 距离范围：0 < distance < 50 英里
- 票价范围：0 < fare < $200
- 时长范围：0 < duration < 180 分钟

然后使用 `StandardScaler` 对三个特征进行标准化：

$$z = \frac{x - \mu}{\sigma}$$

#### 5.3.4 应用场景

- 出行模式聚类散点图（Chart 4）：以距离为X轴、票价为Y轴的散点图，颜色区分簇
- 聚类特征雷达图（Chart 10）：展示各簇在距离、票价、时长、数量四个维度的相对特征

### 5.4 多项式回归预测

#### 5.4.1 方法说明

使用**三次多项式回归**对月度出行量进行拟合预测：

$$y = \beta_0 + \beta_1 x + \beta_2 x^2 + \beta_3 x^3$$

其中 $x$ 为月份（1-12），$y$ 为出行量。

#### 5.4.2 实现细节

| 参数 | 值 | 说明 |
|:---|:---|:---|
| 模型 | LinearRegression | 线性回归（配合多项式特征） |
| 多项式阶数 | degree=3 | 三次多项式 |
| 训练集 | 1-11月 | Jan-Nov |
| 测试集 | 12月 | Dec |
| 评估指标 | MAPE | 平均绝对百分比误差 |

#### 5.4.3 应用场景

- 出行量预测图（Chart 11）：实际值 vs 预测值折线图
- 残差分析图（Chart 12）：各月预测残差分布

### 8.2 24小时分布

- **峰谷比**：最高时与最低时相差约 **7.4倍**
- Green出租车在下午至傍晚时段（15:00-19:00）占比相对更高

## 第十一章 聚类分析

基于K-Means算法，以距离、票价、时长为特征对50,000个采样点进行聚类，识别出 **4种典型出行模式**：

| 聚类 | 占比 | 平均距离 | 平均票价 | 平均时长 | 出行特征 |

| **Cluster 0** | **58.9%** | 1.22 mi | $7.03 | 7.4 min | 短途经济型：曼哈顿区内短途出行 |
| **Cluster 2** | 27.6% | 3.27 mi | $14.72 | 18.2 min | 中程标准型：跨区中距离出行 |
| **Cluster 1** | 10.1% | 7.83 mi | $28.33 | 31.5 min | 长途/机场型：机场接送等 |

## 第十二章 结论与建议

### 12.1 核心发现总结

1. **规模庞大**：全年1.078亿次出行，日均29.5万次，Yellow出租车占92.3%绝对主导
2. **时间双峰**：出行呈早晚双峰分布，晚高峰（18:00）强度是早高峰（8:00）的1.39倍；3月为月度峰值，8月为谷值
3. **空间极度集中**：曼哈顿独占86.7%出行量，Top 5区域全部位于曼哈顿
4. **短途为主**：58.9%的出行为短途经济型（<2英里），中位数距离仅1.76英里
5. **支付电子化**：信用卡已成为主流支付方式（Yellow 70%，Green 57%）
6. **天气敏感**：雪天出行量下降17.4%，雨天仅下降1.7%

### 12.2 运营建议

| 建议 | 说明 |
|:---|:---|
| **动态运力调度** | 根据早晚双峰和月度季节性波动，实施差异化运力配置，重点加强晚高峰时段的车辆投放 |
| **机场专线优化** | 皇后区出行高度依赖机场（拉瓜迪亚+JFK），建议设立机场专线调度机制 |
| **外围区域拓展** | 布朗克斯、布鲁克林出行占比低，可通过Green出租车政策引导提升外围区域服务覆盖 |
| **极端天气预案** | 雪天出行量骤降17.4%，需建立极端天气应急调度方案，保障基本出行需求 |
| **支付方式推广** | Green出租车现金比例仍高达43%，可推广移动支付优惠活动，提升电子支付比例 |
| **票价结构优化** | 短途出行占58.9%，可考虑推出短途优惠套餐，提升高频短途用户的出行体验 |
